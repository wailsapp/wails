Robuste Desktop-Frameworks erfordern ein solides Testen.  
Wails v3 setzt eine **geschichtete Strategie** ein:

| Schicht | Ziel | Tooling |
|-------|------|---------|
| Unit-Tests | Schnelles Feedback für isolierte Funktionen | `go test ./...` |
| Integrations-/Beispieltests | Validierung ganzer Funktionen über Pakete hinweg | Beispiel-Apps + Headless-Assertions |
| Race-Erkennung | Auffinden von Data Races in der Laufzeit & Bridge | `go test -race`, `wails3 dev -race` |
| CI-Matrix | Cross-OS-Sicherheit bei jedem PR | GitHub Actions |

Dieses Dokument erklärt, **wo Tests liegen**, **wie man sie ausführt** und **was in CI passiert**.

---

## 1. Verzeichnis-Konventionen

```
v3/
├── internal/.../_test.go     # Unit-Tests für interne Pakete
├── pkg/.../_test.go          # Tests für die öffentliche API
├── examples/...              # Beispiel-Apps dienen auch als Integrationstests
├── tasks/events/...          # Tests für generierten Code, die in CI ausgeführt werden
└── pkg/application/RACE.md   # Anleitung zu Race-Tests & bekannte sichere Unterdrückungen
```

Richtlinien:

* **Halten Sie Unit-Tests neben dem Code** (`foo.go` ↔ `foo_test.go`)
* Verwenden Sie den **Black-Box-Stil** für `pkg/`-Pakete (`package application_test`)
* Integrations-Hilfsfunktionen gehören in `internal/testutil/` (gemeinsame Mocks, Fixtures)

## 2. Unit-Tests

### Schreiben von Tests

```go
func TestColourParseHex(t *testing.T) {
    c, err := colour.Parse("#FF00FF")
    require.NoError(t, err)
    assert.Equal(t, 255, c.R)
}
```

Empfehlungen:

* Verwenden Sie [`stretchr/testify`](https://github.com/stretchr/testify) – bereits als Vendor-Import enthalten.
* Folgen Sie dem **table-driven**-Stil für Verzweigungen.
* Stubben Sie Plattform-Spezifika hinter Build-Tags (`_test_windows.go` usw.), wenn nötig.

### Lokales Ausführen

```bash
cd v3
go test ./... -cover
```

Taskfile-Verknüpfung:

```
task test          # gleich wie oben + vet + lint
```

## 3. Integrations- & Beispieltests

Jeder Ordner unter `v3/examples/` ist eine **in sich geschlossene App**.  
Drei Techniken machen sie zu Tests:

| Technik | Datei | Was wird geprüft |
|-----------|------|----------------|
| `go test ./...` innerhalb des Beispiels | `*_test.go` startet `wails dev` im Headless-Modus | Build & Start |
| CLI-Golden-Tests | `internal/commands/appimage_test.go` | Generierte Artefakt-Hashes |
| Skriptbasiertes E2E | `tasks/events/generate.go` | Generiert Go-Code, der das Beispiel startet und Laufzeit-Logs prüft |

Alles zusammen ausführen mit:

```
task integration
```

> Der Headless-Modus verwendet **Xvfb** auf Linux-Runnern; macOS & Windows starten minimiert.

## 4. Race-Erkennung

Data Races sind in GUI-Laufzeiten fatal.

### Dedizierte Dokumentation

Siehe `v3/pkg/application/RACE.md` für:

* Bekannte harmlose Races und Begründung für deren Unterdrückung
* Wie man Stack-Traces interpretiert, die Cgo-Grenzen überschreiten

### Lokale Race-Suite

```
go test -race ./...
```

oder

```
wails3 dev -race
```

*Letzteres baut die Laufzeit mit Race-Flags neu und startet ein Demo-Fenster.  
Schließen Sie das Fenster; alle Races werden nach dem Beenden ausgegeben.*

CI führt die Race-Suite auf **linux/amd64** (am schnellsten) für jeden PR aus.

## 5. GitHub Actions Workflows

### 5.1 `build-and-test-v3.yml`

Ort: `.github/workflows/build-and-test-v3.yml`

Wichtige Funktionen:

* **Matrix** über `os: [ubuntu-latest, windows-latest, macos-latest]`
* Cacht Go- & npm-Abhängigkeiten für mehr Geschwindigkeit
* Schritte:
  1. `setup-go` @ 1.25
  2. `go vet ./...`
  3. `go test ./... -v -coverprofile=cover.out`
  4. Build CLI + Beispiel-App (`wails3 build -skip-package`)
  5. Upload der Abdeckungsdaten zu Codecov
* Bedingter **Race**-Job auf Ubuntu:
  ```yaml
  - name: Race Detector
    if: matrix.os == 'ubuntu-latest'
    run: go test -race ./...
  ```

### 5.2 Statische Analyse

* `semgrep.yml` – Security/Lint-Scanning (`.github/workflows/semgrep.yml`)
* `qodana.yaml` – JetBrains Qodana Cloud-Inspektion (optional im PR)

### 5.3 Release-Pipeline

`runtime.yml` baut und veröffentlicht den JS-Laufzeit-Code auf npm bei Tag-Push; außerhalb des Testumfangs, aber von der Build-Matrix referenziert.

## 6. Lokale CI-Parität

Führen Sie die **exakte** CI-Aufgabensammlung über Taskfile aus:

```
task ci
```

Es führt aus:

1. `lint` (golangci-lint, govet)
2. `test` (Unit + Race)
3. `integration`
4. `build` (für die Host-Plattform)

## 7. Fehlerbehebung bei fehlerhaften Tests

| Symptom | Wahrscheinliche Ursache | Lösung |
|---------|--------------|-----|
| **Race in `runtime_darwin.go`** | `mainthread.Call` wurde nicht gesperrt | Verwenden Sie den `mainthread.Sync`-Helfer |
| **Windows-Headless hängt** | Konsolenfenster wartet auf Eingabe | Übergeben Sie `-verbose` und stellen Sie sicher, dass keine Dialoge geöffnet sind |
| **CI-Beispielbuild schlägt fehl** | Template wurde geändert, ohne Beispiel-Abhängigkeiten zu erhöhen | `task update-examples` regeneriert Lockfiles |
| **Coverpkg-Fehler** | Integrationstest importiert `main` | Wechseln Sie zu Build-Tags `//go:build integration` |

---

## 8. Hinzufügen neuer Tests

1. **Unit** – erstellen Sie `*_test.go`, führen Sie `go test ./...` aus
2. **Integration** – aktualisieren oder fügen Sie Beispiel-App + Test-Harness hinzu
3. **CI** – committen; der Workflow entdeckt Tests automatisch  
   Bei OS-spezifischen Tests, fügen Sie Skip-Tags hinzu:

```go
//go:build !windows
```

Qualität ist in Wails v3 kein nachträglicher Gedanke.  
Mit Unit-Tests, Integrationssuites, Race-Erkennung und einer plattformübergreifenden CI-Matrix können Sie selbstbewusst beitragen, in der Gewissheit, dass Ihre Änderungen auf jeder unterstützten OS-Plattform grün durchlaufen.  
Viel Spaß beim Testen!
