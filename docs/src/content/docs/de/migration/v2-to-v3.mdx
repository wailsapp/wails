Wails v3 ist eine **komplette Neuschreibung** mit erheblichen Verbesserungen in Architektur, Leistung und Entwicklererlebnis. Dieser Leitfaden hilft Ihnen, Ihre v2-Anwendung zu v3 zu migrieren.

**Wichtige Änderungen:**
- Neue Anwendungsstruktur
- Verbessertes Bindungssystem
- Erweitertes Fenstermanagement
- Besseres Ereignissystem
- Vereinfachte Konfiguration

**Migrationszeit:** 1-4 Stunden für typische Anwendungen

## Breaking Changes

### Anwendungsinitialisierung

In v2 waren Anwendungssetup, Fensterkonfiguration und Ausführung in einem einzigen `wails.Run()`-Aufruf kombiniert. Dieser monolithische Ansatz erschwerte das Erstellen mehrerer Fenster, das Behandeln von Fehlern in verschiedenen Phasen oder das Testen einzelner Komponenten Ihrer Anwendung.

v3 trennt diese Anliegen in verschiedene Phasen: Anwendungserstellung, Fenstererstellung und Ausführung. Diese Trennung gibt Ihnen explizite Kontrolle über jede Phase des Lebenszyklus Ihrer Anwendung und macht den Code modularer und testbarer.

**v2:**

```go
err := wails.Run(&options.App{
    Title:  "My App",
    Width:  1024,
    Height: 768,
    Bind: []interface{}{
        &GreetService{},
    },
})
```

**v3:**

```go
app := application.New(application.Options{
    Name: "My App",
    Services: []application.Service{
        application.NewService(&GreetService{}),
    },
})

window := app.Window.NewWithOptions(application.WebviewWindowOptions{
    Title:  "My App",
    Width:  1024,
    Height: 768,
})

app.Run()
```

**Warum das besser ist:**

- **Multi-Fenster-Unterstützung**: Sie können Fenster dynamisch zu jedem Zeitpunkt erstellen, nicht nur beim Start
- **Bessere Fehlerbehandlung**: Jede Phase kann separat validiert werden mit angemessener Fehlerbehandlung
- **Klarerer Code**: Die Trennung macht deutlich, was in jeder Phase geschieht
- **Besser testbar**: Sie können das Anwendungssetup testen, ohne die Event-Schleife auszuführen
- **Flexibler**: Fenster können während des gesamten Lebenszyklus der Anwendung erstellt, zerstört und neu erstellt werden

### Bindungen

In v2 erforderte jedes gebundene Struct ein Context-Feld und eine `startup(ctx)`-Methode, um den Runtime-Context zu empfangen. Dies führte zu einer engen Kopplung zwischen Ihrer Geschäftslogik und der Wails-Runtime, was den Code schwieriger zu testen und zu verstehen machte.

v3 führt das Service-Muster ein, bei dem Ihre Structs vollständig eigenständig sind und keinen Runtime-Context speichern müssen. Wenn ein Service Zugriff auf die Anwendungsinstanz benötigt, erhält er diese explizit durch Dependency Injection statt durch implizites Context-Threading.

**v2:**

```go
type App struct {
    ctx context.Context
}

func (a *App) startup(ctx context.Context) {
    a.ctx = ctx
}

func (a *App) Greet(name string) string {
    return "Hello " + name
}
```

**v3:**

```go
type GreetService struct{}

func (g *GreetService) Greet(name string) string {
    return "Hello " + name
}

// Als Service registrieren
app := application.New(application.Options{
    Services: []application.Service{
        application.NewService(&GreetService{}),
    },
})
```

**Warum das besser ist:**

- **Keine impliziten Abhängigkeiten**: Services sind einfache Go-Schnittstellen ohne versteckte Runtime-Abhängigkeiten
- **Einfacheres Testen**: Sie können Service-Methoden testen, ohne einen Wails-Context zu mocken
- **Klarerer Code**: Abhängigkeiten sind explizit (als Konstruktorargumente übergeben) statt in einem Context-Feld versteckt
- **Bessere Organisation**: Services können nach Domäne gruppiert werden, anstatt alle in einem einzigen `App`-Struct zu leben
- **Ordentliche Initialisierung**: Verwenden Sie die `ServiceStartup()`-Methode, wenn Sie Initialisierung benötigen, um dies explizit zu machen

### Runtime

In v2 erforderten alle Runtime-Operationen das Übergeben eines Contexts an globale Funktionen aus dem `runtime`-Paket. Dies führte zu einer engen Kopplung an das Context-Objekt in Ihrem gesamten Codebase und ließ die API prozedural statt objektorientiert wirken.

v3 ersetzt den Context-basierten Runtime durch direkte Methodenaufrufe auf Anwendungs- und Fensterobjekten. Operationen werden direkt auf den Objekten aufgerufen, die sie betreffen, was den Code intuitiver und objektorientierter macht.

**v2:**

```go
import "github.com/wailsapp/wails/v2/pkg/runtime"

runtime.WindowSetTitle(a.ctx, "New Title")
runtime.EventsEmit(a.ctx, "event-name", data)
```

**v3:**

```go
// App-Referenz speichern
type MyService struct {
    app *application.Application
}

func (s *MyService) UpdateTitle() {
    window := s.app.Window.Current()
    window.SetTitle("New Title")
}

func (s *MyService) EmitEvent() {
    s.app.Event.Emit("event-name", data)
}
```

**Warum das besser ist:**

- **Objektorientiertes Design**: Methoden werden auf den Objekten aufgerufen, die sie betreffen (Fenster, App, Menü, usw.)
- **Klarere Absicht**: `window.SetTitle()` ist offensichtlicher als `runtime.WindowSetTitle(ctx, ...)`
- **Bessere IDE-Unterstützung**: Autovervollständigung funktioniert richtig, wenn Methoden auf Objekten sind
- **Klarheit bei Multi-Fenstern**: Bei mehreren Fenstern wählen Sie explizit, welches Fenster Sie bedienen möchten
- **Kein Context-Threading**: Sie müssen keinen Context durch jede Funktion übergeben

### Frontend-Bindungen

In v2 wurden Bindungen nach Go-Paket und Struct-Name organisiert, was typischerweise zu Pfaden wie `wailsjs/go/main/App` führte. Diese Struktur spiegelte keine logische Gruppierung wider und machte es schwierig, verwandte Funktionalität zu finden.

v3 organisiert Bindungen nach Service-Name und Anwendungsmodul, was eine klarere logische Struktur schafft. Die Bindungen werden in ein `bindings`-Verzeichnis generiert, das nach Ihrem Anwendungsnamen und Service-Namen organisiert ist, was es einfacher macht zu verstehen, welche Funktionalität verfügbar ist.

**v2:**

```javascript
import { Greet } from '../wailsjs/go/main/App'

const result = await Greet("World")
```

**v3:**

```javascript
import { Greet } from './bindings/myapp/greetservice'

const result = await Greet("World")
```

**Warum das besser ist:**

- **Logische Organisation**: Bindungen werden nach Service-Name gruppiert statt nach Go-Paket-Struktur
- **Klarere Imports**: Der Pfad spiegelt die Domänenlogik (greetservice) wider, nicht die Dateistruktur (main/App)
- **Bessere Auffindbarkeit**: Sie können Bindungen nach Feature navigieren statt nach technischer Struktur
- **Konsistente Benennung**: Service-basierte Organisation entspricht Ihrer Backend-Architektur
- **Einfachere Pfade**: Kein `../wailsjs/go`-Präfix mehr - nur `./bindings`

### Ereignisse

In v2 verwendeten Ereignisse variadische `interface{}`-Parameter und erforderten das Übergeben eines Contexts an jede Event-Funktion. Event-Handler empfingen untypisierte Daten, die manuelle Typumwandlungen benötigten, was das Ereignissystem fehleranfällig und schwer zu debuggen machte.

v3 führt typisierte Event-Objekte ein und entfernt die Context-Anforderung. Event-Handler empfangen ein ordentliches Event-Objekt mit typisierten Daten, was das Ereignissystem zuverlässiger und einfacher zu verwenden macht.

**v2:**

```go
runtime.EventsOn(ctx, "event-name", func(data ...interface{}) {
    // Event verarbeiten
})

runtime.EventsEmit(ctx, "event-name", data)
```

**v3:**

```go
app.Event.On("event-name", func(e *application.CustomEvent) {
    data := e.Data
    // Event verarbeiten
})

app.Event.Emit("event-name", data)
```

**Warum das besser ist:**

- **Typsicherheit**: Ereignisse verwenden ordentliche Event-Objekte statt `...interface{}`
- **Besseres Debugging**: Event-Objekte enthalten Metadaten wie den Event-Namen, was das Debugging erleichtert
- **Klarere API**: `app.Event.On()` und `app.Event.Emit()` sind intuitiver als Runtime-Funktionen
- **Kein Context erforderlich**: Ereignisse funktionieren direkt auf dem App-Objekt ohne Context-Threading
- **Einfachere Handler**: Event-Handler haben eine klare Signatur statt variadischer Parameter### Windows

v2 unterstützte nur ein einzelnes Fenster pro Anwendung. Das Fenster wurde beim Start erstellt und alle Fensteroperationen wurden über Runtime-Funktionen ausgeführt, die implizit dieses einzelne Fenster ansprachen.

v3 führt native Multi-Window-Unterstützung als Kernfunktion ein. Jedes Fenster ist ein First-Class-Objekt mit eigenen Methoden und Lebenszyklus. Sie können mehrere Fenster dynamisch während der gesamten Lebensdauer Ihrer Anwendung erstellen, verwalten und zerstören.

**v2:**

```go
// Single window only
runtime.WindowSetSize(ctx, 800, 600)
```

**v3:**

```go
// Multiple windows supported
window1 := app.Window.New()
window1.SetSize(800, 600)

window2 := app.Window.New()
window2.SetSize(1024, 768)
```

**Warum das besser ist:**

- **Multi-Window-Anwendungen**: Erstellen Sie Apps mit mehreren unabhängigen Fenstern (Dashboards, Einstellungen, Tools usw.)
- **Explizite Fensterreferenzen**: Jedes Fenster ist ein Objekt, das Sie speichern und direkt manipulieren können
- **Dynamische Fenstererstellung**: Erstellen und zerstören Sie Fenster jederzeit während der Laufzeit
- **Unabhängiger Fensterzustand**: Jedes Fenster hat seine eigenen Ereignisse, Eigenschaften und Lebenszyklus
- **Bessere Architektur**: Das Fenstermanagement ist objektorientiert statt kontextbasiert

## Migrationsschritte

### Schritt 1: Abhängigkeiten aktualisieren

**go.mod:**

```go
module myapp

go 1.21

require (
    github.com/wailsapp/wails/v3 v3.0.0-alpha.1
)
```

**Aktualisieren:**

```bash
go get github.com/wailsapp/wails/v3@latest
go mod tidy
```

### Schritt 2: main.go aktualisieren

**v2:**

```go
package main

import (
    "embed"
    "github.com/wailsapp/wails/v2/pkg/options"
    "github.com/wailsapp/wails/v2/pkg/options/assetserver"
    "github.com/wailsapp/wails/v2/pkg/options/windows"
)

//go:embed all:frontend/dist
var assets embed.FS

func main() {
    app := NewApp()

    err := wails.Run(&options.App{
        Title:  "My App",
        Width:  1024,
        Height: 768,
        AssetServer: &assetserver.Options{
            Assets: assets,
        },
        Bind: []interface{}{
            app,
        },
        Windows: &windows.Options{
            WebviewIsTransparent: false,
        },
    })

    if err != nil {
        println("Error:", err.Error())
    }
}
```

**v3:**

```go
package main

import (
    "embed"
    "github.com/wailsapp/wails/v3/pkg/application"
)

//go:embed frontend/dist
var assets embed.FS

func main() {
    app := application.New(application.Options{
        Name: "My App",
        Services: []application.Service{
            application.NewService(&MyService{}),
        },
        Assets: application.AssetOptions{
            Handler: application.AssetFileServerFS(assets),
        },
    })

    app.Window.NewWithOptions(application.WebviewWindowOptions{
        Title:  "My App",
        Width:  1024,
        Height: 768,
    })

    err := app.Run()
    if err != nil {
        panic(err)
    }
}
```

### Schritt 3: App-Struktur in Service konvertieren

**v2:**

```go
type App struct {
    ctx context.Context
}

func NewApp() *App {
    return &App{}
}

func (a *App) startup(ctx context.Context) {
    a.ctx = ctx
    // Initialisation
}

func (a *App) Greet(name string) string {
    return "Hello " + name
}
```

**v3:**

```go
type MyService struct {
    app *application.Application
}

func NewMyService(app *application.Application) *MyService {
    return &MyService{app: app}
}

func (s *MyService) ServiceStartup(ctx context.Context, options application.ServiceOptions) error {
    // Initialisation
    return nil
}

func (s *MyService) Greet(name string) string {
    return "Hello " + name
}

// Register after app creation
app := application.New(application.Options{})
app.RegisterService(application.NewService(NewMyService(app)))
```

### Schritt 4: Runtime-Aufrufe aktualisieren

**v2:**

```go
func (a *App) DoSomething() {
    runtime.WindowSetTitle(a.ctx, "New Title")
    runtime.EventsEmit(a.ctx, "update", data)
    runtime.LogInfo(a.ctx, "Message")
}
```

**v3:**

```go
func (s *MyService) DoSomething() {
    window := s.app.Window.Current()
    window.SetTitle("New Title")
    
    s.app.Event.Emit("update", data)
    
    s.app.Logger.Info("Message")
}
```

### Schritt 5: Frontend aktualisieren

**Neue Bindings generieren:**

```bash
wails3 generate bindings
```

**Imports aktualisieren:**

```javascript
// v2
import { Greet } from '../wailsjs/go/main/App'

// v3
import { Greet } from './bindings/myapp/myservice'
```

**Ereignisbehandlung aktualisieren:**

```javascript
// v2
import { EventsOn, EventsEmit } from '../wailsjs/runtime/runtime'

EventsOn("update", (data) => {
    console.log(data)
})

EventsEmit("action", data)

// v3
import { Events } from '@wailsio/runtime'

Events.On("update", (data) => {
    console.log(data)
})

Events.Emit("action", data)
```

## Feature-Zuordnung

### Dialoge

**v2:**

```go
selection, err := runtime.OpenFileDialog(ctx, runtime.OpenDialogOptions{
    Title: "Select File",
})
```

**v3:**

```go
selection, err := app.Dialog.OpenFile(application.OpenFileDialogOptions{
    Title: "Select File",
})
```

### Menüs

**v2:**

```go
menu := menu.NewMenu()
menu.Append(menu.Text("File", nil, []*menu.MenuItem{
    menu.Text("Quit", nil, func(_ *menu.CallbackData) {
        runtime.Quit(ctx)
    }),
}))
```

**v3:**

```go
menu := app.NewMenu()
fileMenu := menu.AddSubmenu("File")
fileMenu.Add("Quit").OnClick(func(ctx *application.Context) {
    app.Quit()
})
```

### System Tray

**v2:**

```go
// Not available in v2
```

**v3:**

```go
systray := app.SystemTray.New()
systray.SetIcon(iconBytes)
systray.SetLabel("My App")

menu := app.NewMenu()
menu.Add("Show").OnClick(showWindow)
menu.Add("Quit").OnClick(app.Quit)
systray.SetMenu(menu)
```

## Häufige Probleme

### Problem: Bindings nicht gefunden

**Problem:** Importfehler nach der Migration

**Lösung:**

```bash
# Regenerate bindings
wails3 generate bindings

# Check output directory
ls frontend/bindings
```

### Problem: Kontextfehler

**Problem:** `ctx` nicht verfügbar

**Lösung:**

Speichern Sie die App-Referenz stattdessen:

```go
type MyService struct {
    app *application.Application
}

func NewMyService(app *application.Application) *MyService {
    return &MyService{app: app}
}
```

### Problem: Fenstermethoden funktionieren nicht

**Problem:** `runtime.WindowSetTitle()` existiert nicht

**Lösung:**

Verwenden Sie Fenstermethoden direkt:

```go
window := s.app.Window.Current()
window.SetTitle("New Title")
```

### Problem: Ereignisse werden nicht ausgelöst

**Problem:** Ereignisse registriert, aber nicht empfangen

**Lösung:**

Überprüfen Sie, ob die Ereignisnamen exakt übereinstimmen:

```go
// Go
app.Event.Emit("my-event", data)

// JavaScript
OnEvent("my-event", handler)  // Must match exactly
```

## Vorteile von v3

### Leistung

- **Schnellerer Start** - Optimierte Initialisierung
- **Geringerer Speicherbedarf** - Effiziente Ressourcennutzung
- **Bessere Bridge** - &lt;1ms Aufruf-Overhead

### Funktionen

- **Multi-Window** - Native Unterstützung
- **System Tray** - Eingebaut
- **Bessere Ereignisse** - Typisiert, einfachere API
- **Services** - Bessere Code-Organisation

### Entwicklererfahrung

- **Typsicherheit** - Vollständige TypeScript-Unterstützung
- **Bessere Fehlermeldungen** - Klare Fehlermeldungen
