## Paketformate

Paketieren Sie Ihre App für die Linux-Verteilung:

```bash
wails3 package GOOS=linux
```

Dies erstellt mehrere Formate im Verzeichnis `bin/`:
- **AppImage**: Tragbar, läuft auf jeder Linux-Distribution
- **DEB**: Für Debian, Ubuntu und Derivate
- **RPM**: Für Fedora, RHEL und Derivate
- **Arch**: Für Arch Linux und Derivate

### Einzelne Formate

Erstellen Sie spezifische Formate:

```bash
wails3 task linux:create:appimage
wails3 task linux:create:deb
wails3 task linux:create:rpm
wails3 task linux:create:aur
```

## Anpassen von Paketen

### Desktop-Eintrag

Die `.desktop`-Datei steuert, wie Ihre App in den Anwendungsmenüs erscheint. Sie wird aus Werten in `build/linux/Taskfile.yml` generiert:

```yaml
vars:
  APP_NAME: 'MyApp'
  EXEC: 'MyApp'
  ICON: 'MyApp'
  CATEGORIES: 'Development;'
```

### Paketmetadaten

Bearbeiten Sie `build/linux/nfpm/nfpm.yaml`, um DEB- und RPM-Pakete anzupassen:

```yaml
name: myapp
version: 1.0.0
maintainer: Ihr Name <you@example.com>
description: Meine großartige Wails-Anwendung
homepage: https://example.com
license: MIT
```

### AppImage

Die AppImage-Konfiguration befindet sich in `build/linux/appimage/`. Das App-Symbol stammt aus `build/appicon.png`.

## Pakete signieren

Signieren Sie DEB- und RPM-Pakete mit einem PGP-Schlüssel:

```bash
# Mit dem Wrapper (erkennt die Plattform automatisch)
wails3 sign GOOS=linux

# Oder direkt mit Tasks
wails3 task linux:sign:deb
wails3 task linux:sign:rpm
wails3 task linux:sign:packages  # Beide
```

Konfigurieren Sie die Signatur in `build/linux/Taskfile.yml`:

```yaml
vars:
  PGP_KEY: "path/to/signing-key.asc"
  SIGN_ROLE: "builder"  # origin, maint, archive oder builder
```

Speichern Sie Ihr Schlüssel-Passwort:

```bash
wails3 setup signing
```

Weitere Details finden Sie unter [Anwendungen signieren](/de/guides/build/signing).

## Bauen für ARM

```bash
wails3 build GOOS=linux GOARCH=arm64
wails3 package GOOS=linux GOARCH=arm64
```

<Aside type="note">
ARM64-Builds von x86_64-Hosts verwenden Docker für die CGO-Cross-Kompilierung.
</Aside>

## GTK4-Unterstützung (Experimentell)

Wails v3 enthält experimentelle Unterstützung für GTK4 mit WebKitGTK 6.0. Standardmäßig verwendet Wails GTK3 mit WebKit2GTK 4.1. Die GTK4-Unterstützung ist optional und muss über einen Build-Tag aktiviert werden.

<Aside type="caution" title="Experimentell">
Die GTK4-Unterstützung ist experimentell und kann Probleme aufweisen. Bitte melden Sie Probleme im [GitHub Issue #4957](https://github.com/wailsapp/wails/issues/4957).
</Aside>

### Abhängigkeiten

Installieren Sie die GTK4- und WebKitGTK 6.0-Entwicklungsbibliotheken:

```bash
# Ubuntu/Debian
sudo apt install libgtk-4-dev libwebkitgtk-6.0-dev

# Fedora
sudo dnf install gtk4-devel webkitgtk6.0-devel

# Arch
sudo pacman -S gtk4 webkitgtk-6.0
```

Die erforderlichen pkg-config-Pakete sind `gtk4` und `webkitgtk-6.0`.

### Bauen mit GTK4

Verwenden Sie die Option `-tags gtk4`:

```bash
wails3 build -tags gtk4
```

Oder direkt mit Go:

```bash
go build -tags gtk4 -o myapp .
```

### Bekannte Unterschiede zu GTK3

- **Dateidialoge**: GTK4 verwendet `xdg-desktop-portal` für Dateidialoge, was bedeutet, dass einige Dialogoptionen (wie das Standardverzeichnis, die Anzeige benutzerdefinierter Filter) anders funktionieren. Weitere Details finden Sie unter [Dialoge-Referenz - Linux GTK4-Einschränkungen](/reference/dialogs#linux-gtk4-limitations).
- **Menüstil**: GTK4 unterstützt eine Option `LinuxMenuStylePrimaryMenu`, die einen Hamburger-Button (☰) in der Header-Leiste anzeigt, entsprechend den GNOME HIG. Siehe [Fenster-API - Linux MenuStyle](/reference/window#linux).
- **DPI-Skalierung**: GTK4 verwendet `gdk_monitor_get_scale` (GTK 4.14+) für die Unterstützung der gebrochenzahligen Skalierung.

### Überprüfen Ihres Builds

Führen Sie `wails3 doctor` aus, um Ihre Einrichtung zu überprüfen. GTK4-Abhängigkeiten werden als optional/experimentell angezeigt.

## Fehlerbehebung

### AppImage lässt sich nicht ausführen

Machen Sie es ausführbar:

```bash
chmod +x MyApp-x86_64.AppImage
```

### Fehlende Abhängigkeiten

Wenn die App nicht startet, prüfen Sie auf fehlende WebKit-Abhängigkeiten:

```bash
# Debian/Ubuntu
sudo apt install libwebkit2gtk-4.1-0

# Fedora
sudo dnf install webkit2gtk4.1

# Arch
sudo pacman -S webkit2gtk-4.1
```

### Kein C-Compiler gefunden

Das Build-System benötigt GCC oder Clang für CGO:

```bash
# Debian/Ubuntu
sudo apt install build-essential

# Fedora
sudo dnf install gcc

# Arch
sudo pacman -S base-devel
```

Alternativ können Sie `wails3 task setup:docker` ausführen, und das Build-System verwendet automatisch Docker.

### Leeres oder weißes Fenster auf NVIDIA-GPU

Auf Linux mit proprietären NVIDIA-Treibern können Wails-Apps beim Start ein leeres oder weißes Fenster anzeigen. Dies wird durch einen WebKitGTK-Bug verursacht, bei dem der DMA-BUF-Renderer mit `gbm_bo_map()` gegen den proprietären NVIDIA-Treiber fehlschlägt (betrifft X11 und Wayland, Treiberversionen 377–580+, GPUs ab der 10er-Serie und ältere GT 710).

**Wails setzt `WEBKIT_DISABLE_DMABUF_RENDERER=1` automatisch**, wenn es das NVIDIA-Kernelmodul (`/sys/module/nvidia`) erkennt, sodass die meisten Benutzer nichts tun müssen.

Wenn Sie dennoch ein leeres Fenster sehen (z. B. in einem Container, wo der Modulpfad nicht sichtbar ist), setzen Sie die Umgebungsvariable manuell, bevor Sie Ihre App starten:

```bash
WEBKIT_DISABLE_DMABUF_RENDERER=1 ./myapp
```

Verwandte Upstream-Bugs: [WebKit #262607](https://bugs.webkit.org/show_bug.cgi?id=262607), [WebKit #180739](https://bugs.webkit.org/show_bug.cgi?id=180739).

### AppImage-Strip-Kompatibilität

Auf modernen Linux-Distributionen (Arch Linux, Fedora 39+, Ubuntu 24.04+) werden Systembibliotheken mit `.relr.dyn`-ELF-Abschnitten für effizientere Relokationen kompiliert. Das Tool `linuxdeploy`, das zum Erstellen von AppImages verwendet wird, bündelt eine ältere `strip`-Binärdatei, die diese modernen Abschnitte nicht verarbeiten kann.

Wails erkennt diese Situation automatisch, indem es die System-GTK-Bibliotheken vor dem Erstellen des AppImages überprüft. Wenn dies erkannt wird, wird das Strippen deaktiviert (`NO_STRIP=1`), um die Kompatibilität sicherzustellen.

**Was das bedeutet:**
- AppImages sind auf betroffenen Systemen etwas größer (~20-40%)
- Die Funktionsfähigkeit der Anwendung wird nicht beeinträchtigt
- Dies wird automatisch gehandhabt—keine Aktion erforderlich

Wenn Sie kleinere AppImages auf modernen Systemen benötigen, können Sie eine neuere `strip`-Binärdatei installieren und `linuxdeploy` so konfigurieren, dass sie anstelle der gebündelten Version verwendet wird.
