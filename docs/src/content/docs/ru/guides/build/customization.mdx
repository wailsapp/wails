## Обзор

Система сборки Wails — это гибкий и мощный инструмент, предназначенный для упрощения
процесса сборки ваших приложений Wails. Она использует
[Task](https://taskfile.dev), инструмент для выполнения задач, который позволяет легко
определять и запускать задачи. Хотя система сборки v3 является стандартной, Wails
поощряет подход «используйте свои собственные инструменты», позволяя разработчикам
настраивать процесс сборки по мере необходимости.

Узнайте больше о том, как использовать Task, в
[официальной документации](https://taskfile.dev/usage/).

## Task: Сердце системы сборки

[Task](https://taskfile.dev) — это современная альтернатива Make, написанная на Go. Он
использует YAML-файл для определения задач и их зависимостей. В системе сборки Wails
[Task](https://taskfile.dev) играет центральную роль в координации
процесса сборки.

Основной файл `Taskfile.yml` находится в корне проекта, а задачи, специфичные для
платформ, определены в файлах `build/<platform>/Taskfile.yml`. Общий файл `Taskfile.yml`
в директории `build` содержит общие задачи, которые используются на всех
платформах.

## Taskfile.yml

Файл `Taskfile.yml` в корне проекта является основной точкой входа для системы сборки. Он определяет
задачи и их зависимости. Вот стандартный файл `Taskfile.yml`:

```yaml
version: '3'

includes:
  common: ./build/Taskfile.yml
  windows: ./build/windows/Taskfile.yml
  darwin: ./build/darwin/Taskfile.yml
  linux: ./build/linux/Taskfile.yml

vars:
  APP_NAME: "myproject"
  BIN_DIR: "bin"
  VITE_PORT: '{{.WAILS_VITE_PORT | default 9245}}'

tasks:
  build:
    summary: Builds the application
    cmds:
      - task: "{{OS}}:build"

  package:
    summary: Packages a production build of the application
    cmds:
      - task: "{{OS}}:package"

  run:
    summary: Runs the application
    cmds:
      - task: "{{OS}}:run"

  dev:
    summary: Runs the application in development mode
    cmds:
      - wails3 dev -config ./build/config.yml -port {{.VITE_PORT}}


```

## Платформоспецифичные Taskfile

У каждой платформы есть свой Taskfile, расположенный в директориях платформ внутри директории `build`. Эти
файлы определяют основные задачи для данной платформы. Каждый taskfile включает общие задачи из файла `build/Taskfile.yml`.

### Windows

Расположение: `build/windows/Taskfile.yml`

Windows-специфичный Taskfile включает задачи для сборки, упаковки и
запуска приложения на Windows. Ключевые особенности:

- Сборка с необязательными флагами для продакшена
- Генерация иконки в формате `.ico`
- Генерация файла `.syso` для Windows
- Создание установщика NSIS для упаковки

### Linux

Расположение: `build/linux/Taskfile.yml`

Linux-специфичный Taskfile включает задачи для сборки, упаковки и запуска
приложения на Linux. Ключевые особенности:

- Сборка с необязательными флагами для продакшена
- Создание пакетов AppImage, deb, rpm и Arch Linux
- Генерация файла `.desktop` для Linux-приложений

### macOS

Расположение: `build/darwin/Taskfile.yml`

macOS-специфичный Taskfile включает задачи для сборки, упаковки и запуска
приложения на macOS. Ключевые особенности:

- Сборка бинарных файлов для архитектур amd64, arm64 и универсальной (обе)
- Генерация иконки в формате `.icns`
- Создание пакета `.app` для распространения
- Подписание пакетов `.app` ad-hoc
- Установка специфичных для macOS флагов сборки и переменных окружения

## Выполнение задач и алиасы команд

Команда `wails3 task` — это встроенная версия [Taskfile](https://taskfile.dev), которая выполняет
задачи, определенные в вашем `Taskfile.yml`.

Команды `wails3 build` и `wails3 package` являются алиасами для
`wails3 task build` и `wails3 task package` соответственно. Когда вы запускаете эти
команды, Wails внутренне преобразует их в соответствующее выполнение задач:

- `wails3 build` → `wails3 task build`
- `wails3 package` → `wails3 task package`

### Передача параметров задачам

Вы можете передавать переменные CLI задачам в формате `KEY=VALUE`. Эти переменные передаются через команды-алиасы:

```bash
# Эти команды эквивалентны:
wails3 build PLATFORM=linux CONFIG=production
wails3 task build PLATFORM=linux CONFIG=production

# Упаковка с пользовательской версией:
wails3 package VERSION=2.0.0 OUTPUT=myapp.pkg
```

В вашем `Taskfile.yml` вы можете получить доступ к этим переменным, используя синтаксис Go-шаблонов:

```yaml
tasks:
  build:
    cmds:
      - echo "Building for {{.PLATFORM | default "darwin"}}"
      - go build -tags {{.CONFIG | default "debug"}} -o myapp
```

## Общий процесс сборки

На всех платформах процесс сборки обычно включает следующие шаги:

1. Обновление Go-модулей
2. Сборка фронтенда
3. Генерация иконок
4. Компиляция Go-кода с платформоспецифичными флагами
5. Упаковка приложения (платформоспецифично)

## Настройка процесса сборки

Хотя система сборки v3 предоставляет надежную конфигурацию по умолчанию, вы можете легко
настроить её под потребности вашего проекта. Изменяя `Taskfile.yml` и
платформоспецифичные Taskfile, вы можете:

- Добавлять новые задачи
- Изменять существующие задачи
- Менять порядок выполнения задач
- Интегрироваться с другими инструментами и скриптами

Эта гибкость позволяет вам адаптировать процесс сборки под ваши конкретные
требования, по-прежнему пользуясь структурой, предоставляемой системой сборки Wails.

:::tip[Изучение Taskfile]
Мы настоятельно рекомендуем прочитать документацию [Taskfile](https://taskfile.dev), чтобы
понять, как эффективно использовать Taskfile.
Вы можете узнать, какая версия Taskfile встроена в CLI Wails, запустив `wails3 task --version`.
:::

## Режим разработки

Система сборки Wails включает мощный режим разработки, который улучшает
опыт разработчика за счет обеспечения горячей перезагрузки и замены модулей.
Этот режим активируется с помощью команды `wails3 dev`.

### Как это работает

Когда вы запускаете `wails3 dev`, происходит следующее:

1. Команда проверяет доступный порт, по умолчанию используется 9245, если не
   указано иное.
2. Она устанавливает переменные окружения для сервера разработки фронтенда (Vite).
3. Она запускает наблюдатель за файлами с помощью библиотеки [refresh](https://github.com/atterpac/refresh).

Библиотека [refresh](https://github.com/atterpac/refresh) отвечает за
мониторинг изменений файлов и запуск пересборки. Она использует конфигурацию, определенную в ключе `dev_mode` в файле `./build/config.yaml`.
Можно настроить игнорирование определенных директорий и файлов, чтобы определить, какие файлы отслеживать и какие действия выполнять при обнаружении изменений.
Конфигурация по умолчанию работает довольно хорошо, но вы можете настроить её под свои нужды.

### Конфигурация

Вот пример её структуры:

```yaml
dev_mode:
  root_path: .
  log_level: warn
  debounce: 1000
  ignore:
    dir:
      - .git
      - node_modules
      - frontend
      - bin
    file:
      - .DS_Store
      - .gitignore
      - .gitkeep
    watched_extension:
      - "*.go"
    git_ignore: true
  executes:
    - cmd: wails3 task common:install:frontend:deps
      type: once
    - cmd: wails3 task common:dev:frontend
      type: background
    - cmd: go mod tidy
      type: blocking
    - cmd: wails3 task build
      type: blocking
    - cmd: wails3 task run
      type: primary
```

Этот конфигурационный файл позволяет вам:

- Установить корневой путь для отслеживания файлов
- Настроить уровень логирования
- Установить время задержки для событий изменения файлов
- Игнорировать определенные директории, файлы или расширения файлов
- Определить команды для выполнения при изменении файлов### Настройка режима разработки

Вы можете настроить режим разработки, изменив следующие значения в файле `config.yaml`.

Некоторые способы настройки включают:

1. Изменение отслеживаемых директорий или файлов
2. Настройку времени задержки (debounce time) для управления скоростью реакции системы на изменения
3. Добавление или изменение команд выполнения в соответствии с потребностями вашего проекта

### Использование браузера для разработки

Хотя Wails v2 полностью поддерживал использование браузера для разработки, это вызывало много путаницы. Приложения, которые работали в браузере, не обязательно работали бы в десктопном приложении, поскольку не все браузерные API доступны в WebView.

Для разработки, ориентированной на UI, в v3 вы по-прежнему можете использовать браузер, открыв URL Vite по адресу `http://localhost:9245` в режиме разработки. Это предоставляет доступ к мощным инструментам разработки браузера при работе со стилями и макетом. Имейте в виду, что привязки Go *не будут работать* в этом режиме.
Когда вы будете готовы протестировать функциональность, такую как привязки и события, просто переключитесь на просмотр десктопного приложения, чтобы убедиться, что всё работает идеально в производственной среде.
