`wails3 build` — это **одна команда**, которая запускает _многоэтапный_ конвейер:

1. **Производственная сборка фронтенда** (Vite / React / …)  
2. **Встраивание ресурсов** в исходный код Go  
3. **Нативная компиляция** для целевой ОС/архитектуры  
4. **Постобработка** (внедрение иконок, информации о версии, кодоподпись)  
5. **Упаковка** в распространяемый артефакт (AppImage, DMG, MSI, …)

В этом документе описаны все этапы, показано, где находится код, и объяснено,
как его настроить или отладить.

---

## 1. Точка входа: `internal/commands/build-assets.go`

```
wails3 build                 # cmd/wails3/main.go
└─ internal/commands.Build() # build-assets.go
```

Высокоуровневые задачи делегируются целям **Taskfile**, чтобы одна и та же логика
работала как в CI, так и локально.

| Этап | Цель Taskfile | Реализация на Go |
|-------|-----------------|-------------------|
| Сборка фронтенда | `frontend:build` | `internal/commands/task.go` |
| Встраивание ресурсов | `embed:assets`  | сгенерированный `bundled_assetserver.go` |
| Компиляция Go | `build:go`      | `tool_buildinfo.go`, `tool_package.go` |
| Упаковка | `package:*`     | `internal/packager`, `internal/commands/*` |

---

## 2. Производственная сборка фронтенда

1. CLI переходит в директорию `frontend/` и выполняет задачу `build`, найденную в
   файле проекта `Taskfile.yml` (по умолчанию `npm run build`).
2. Результат записывается в `frontend/dist/`.
3. Генерируется манифест с **хэшем содержимого** (`manifest.json`), чтобы
   обход кэша работал из коробки.

Если задача завершается ошибкой, конвейер прерывается досрочно, возвращая код
выхода вашего инструмента сборки.

---

## 3. Встраивание ресурсов

Реализовано в `internal/assetserver/build_production.go`:

* Обходит `frontend/dist` и генерирует файл `go:embed`
  `bundled_assetserver.go` (игнорируется git).
* Добавляет тег сборки `production`.

Результат: финальный бинарный файл содержит каждый байт HTML/JS/CSS, поэтому
исполняемый файл является самодостаточным.

## 4. Нативная компиляция

### Флаги сборки

| Флаг | Назначение |
|------|---------|
| `-tags production` | Выбор prod-сервера ресурсов и заглушек рантайма |
| `-ldflags "-s -w"` | Очистка таблицы символов и DWARF (меньший размер бинарников) |
| `-X internal/buildinfo.BuildTime=$(date)` | Встраивание воспроизводимых метаданных |

`internal/commands/tool_buildinfo.go` собирает версию, коммит git и время сборки,
а затем внедряет их с помощью `go build -ldflags`.

### Матрица кросс-компиляции

| ОС | Архитектура | Тег сборки | CGO | Примечания |
|----|------|-----------|-----|-------|
| Windows | amd64 / arm64 | `windows` | cgo включён для WebView2 | Генерирует `.syso` через `internal/commands/syso.go` |
| macOS   | amd64 / arm64 | `darwin`  | cgo включён (Objective-C) | Доступны задачи кодописи и нотаризации |
| Linux   | amd64 / arm64 | `linux`   | чистый Go (опция webkit)   | Требуются заголовки GTK/WebKitGTK для сборки с CGO |

`wails3 build -platform darwin/arm64` переопределяет GOOS/GOARCH.  
Если CGO нужен на хосте, который не может скомпилировать целевую платформу (например,
сборка Windows из Linux), CLI использует образы **Docker** (`ghcr.io/wailsapp/cross-compiler`).

## 5. Постобработка

### Иконки и ресурсы

* **Windows** – `syso.go` объединяет ваш PNG/ICO в `.syso`, который `go build`
  линкует автоматически. Также записывает `manifest.xml`, включающий поддержку High-DPI.
* **macOS** – `icons.go` преобразует `icon.png` → `.icns`, внедряет в
  `MyApp.app/Contents/Resources`.
* **Linux** – файлы `.desktop` генерируются в `/usr/share/applications`
  каждым бэкендом упаковщика.

### Кодоподпись (опционально)

* macOS: `codesign` + `xcrun altool --notarize-app`
* Windows: `signtool.exe`
* Linux: Не распространено (GPG репозитория обрабатывается внешне)

Цели Taskfile: `sign:mac`, `sign:windows`.

## 6. Бэкенды упаковки

### Linux

| Артефакт | Путь к коду | Инструменты |
|----------|-----------|---------|
| **AppImage** (по умолчанию) | `internal/commands/appimage.go` | `linuxdeploy` + `linuxdeploy-plugin-gtk` |
| **deb / rpm / pacman** | `internal/packager` | `fpm` (вызывается через Go) |

Флаги:

```
wails3 build -package deb   # создаёт myapp_1.0.0_amd64.deb
wails3 build -package rpm
```

### macOS

* **DMG** – `internal/commands/packager.go` вызывает `appdmg` для генерации
  установщика drag-&-drop.
* **ZIP** – запасной вариант, если `appdmg` отсутствует.
* Автоматически устанавливает идентификаторы CFBundle и версию в plist.

### Windows

* **MSI** – `internal/commands/packager.go` оборачивает инструменты **WiX** candle и light.  
  Heat автоматически генерирует дерево компонентов из собранного `.exe`.

Дополнительные ресурсы:

* `internal/commands/windows_resources/` содержит шаблонные фрагменты **.wxs**.
* Информация о версии внедряется через утилиту `rsrc`.

## 7. Структура артефактов сборки

После успешной сборки CLI выводит:

```
build/bin/
├── myapp-linux-amd64
└── myapp-linux-amd64.AppImage

build/package/
└── myapp_1.0.0_amd64.deb
```

Точный набор файлов зависит от флагов `-platform` и `-package`.

## 8. Настройка конвейера

| Потребность | Подход |
|------|----------|
| Запуск линтера перед сборкой | Добавьте задачу `lint` в **Taskfile.yml** и сделайте `build` зависимой от неё. |
| Дополнительные флаги линкера | `wails3 build -ldflags "-H windowsgui"` |
| Пропуск упаковки | `wails3 build -skip-package` (сохраняет сырой бинарный файл). |
| Использование своего упаковщика | Реализуйте `internal/packager/<name>.go`, зарегистрируйте в `packager.go`. |

Все цели Taskfile используют переменные окружения, экспортируемые CLI, поэтому вы можете
использовать значения вроде `$WAILS_VERSION` или `$WAILS_PLATFORM` внутри пользовательских задач.

## 9. Устранение неполадок

| Симптом | Вероятная причина | Решение |
|---------|--------------|-----|
| **`ld: framework not found WebKit` (mac)** | Отсутствуют инструменты CLI Xcode | `xcode-select --install` |
| **Пустое окно в prod-сборке** | Ошибка сборки фронтенда или роутинг SPA | Проверьте, существует ли `frontend/dist/index.html` и установлен ли `History API Fallback`. |
| **`wixl` not found** (кросс-сборка MSI для Linux) | Набор инструментов WiX не установлен в образе Docker | Используйте сборку с `--docker` или установите WiX вручную. |
| **Дублирующийся символ `_WinMain`** | Смешивание флага `windowsgui` и syso | Удалите пользовательские `-ldflags` или позвольте Wails управлять ресурсами. |

Режим подробного вывода: `wails3 build -verbose` выводит каждую выполненную команду.
