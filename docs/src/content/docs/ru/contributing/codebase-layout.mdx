Wails v3 находится в **монорепозитории**, который включает в себя среду выполнения фреймворка, CLI, примеры, документацию и инструменты сборки.  
На этой странице мы рассмотрим *структуру каталогов*, которая важна для тех, кто изучает внутреннее устройство системы.

## Общий обзор

```
wails/
├── v3/               # ⬅️ Всё, что относится к Wails v3, находится здесь
├── v2/               # Устаревшая реализация v2 (можно игнорировать при работе с v3)
├── docs/             # Сайт документации v3 на Astro (эта страница!)
├── website/          # Сайт на Docusaurus v2 и маркетинговые страницы (основной сайт)
├── scripts/          # Различные вспомогательные скрипты (например, генератор изображений спонсоров)
├── mkdocs-website/   # Устаревший прототип документации v3 на mkdocs
└── *.md              # Метафайлы проекта (CHANGELOG, LICENSE, …)
```

Далее мы углубимся в дерево **`v3/`**.

## Корень `v3/`

```
v3/
├── cmd/          # Выполняемые команды (в настоящее время только CLI wails3)
├── examples/     # Практические примеры приложений, тестирующие каждую функцию
├── internal/     # Реализация фреймворка (не публичный API)
├── pkg/          # Публичные пакеты Go — поверхность API
├── tasks/        # Утилиты для релизов на основе Taskfile
├── templates/    # Предложения в стиле RFC (WEP) + общие ресурсы
├── go.mod
└── go.sum
```

### Ментальная модель

1. **`pkg/`** предоставляет *то, что импортируют разработчики приложений*  
2. **`internal/`** содержит *реализацию внутренней логики*  
3. **`cmd/wails3`** управляет *жизненным циклом проекта и сборкой*  
4. **`examples/`** служат одновременно *живыми тестами* и *примерами кода*  

Всё остальное поддерживает эти три основных компонента.

---

## `cmd/` – Команды

| Путь | Примечания |
|------|-------|
| `v3/cmd/wails3` | **Точка входа CLI**. Небольшой `main.go` делегирует всю логику пакетам в `internal/commands`. |
| `internal/commands/*` | Подкоманды (init, dev, build, doctor, …). Каждая находится в отдельном файле для удобства поиска. |
| `internal/commands/task_wrapper.go` | Связывает флаги CLI с конвейером сборки Taskfile. |

CLI отвечает за:

* **Создание структуры проекта** (`init`, генерация шаблонов)  
* **Оркестрацию сервера разработки** (`dev`, live-reload)  
* **Сборку и упаковку для продакшена** (`build`, `package`, обёртки для платформ)  
* **Диагностику** (`doctor`)

## `internal/` – Внутренняя логика

```
internal/
├── assetserver/  # Обслуживание и встраивание веб-ресурсов
├── buildinfo/    # Воспроизводимые метаданные сборки
├── commands/     # Механика CLI (см. выше)
├── runtime/      # Среда выполнения для десктопа (для каждой платформы)
├── generator/    # Статический анализ и генератор привязок
├── templates/    # Шаблоны проектов (фронтенд-стеки)
├── packager/     # Linux AppImage/deb/rpm, Windows MSI, macOS DMG
├── capabilities/ # Проверка возможностей хостовой ОС
├── dbus/         # Интеграция с системным треем Linux
├── service/      # Фреймворк микросервисов IPC
└── ...           # [другие вспомогательные подпакеты]
```

### Ключевые подпакеты

| Пакет | Ответственность | Где подключается |
|---------|----------------|-------------------|
| `runtime` | Окна, цикл событий, буфер обмена, диалоги, системный трей. По одному файлу на ОС с тегами сборки (`*_darwin.go`, `*_linux.go`, …). | Вызывается из `pkg/application` и процессора сообщений. |
| `assetserver` | Сервер файлов в двух режимах:<br />• Dev: обслуживает с диска и проксирует Vite<br />• Prod: встраивает ресурсы через `go:embed` | Инициализируется `pkg/application` при запуске. |
| `generator` | Парсит исходный код Go для создания **метаданных привязок**, которые затем генерируют файлы-заглушки TypeScript и код сериализации/десериализации Go. | Запускается через `wails3 init` / `wails3 generate`. |
| `packager` | Обёртывает платформо-специфичные CLI для упаковки (например, аналоги `electron-builder`) в Go для кроссплатформенной автоматизации. | Вызывается через `wails3 package`. |

Вспомогательные утилиты (например, `s/`, `hash/`, `flags/`) обеспечивают разделение внутренних зависимостей.

## `pkg/` – Публичный API

```
pkg/
├── application/   # Основной API: окна, меню, диалоги, события, …
├── runtime/       # Вспомогательные функции на стороне JS (мост, события, экран, ...)
├── options/       # Строго типизированные структуры конфигурации
├── menu/          # Декларативный DSL для меню
└── ...            # Платформо-специфичные помощники (mac/, windows/, assetserver/, …)
```

`pkg/application` инициализирует приложение Wails:

```go
func main() {
    app := application.New(application.Options{
        Name: "MyApp",
        Assets: application.NewAssetsFS(assetsFS),
    })
    window := app.Window.New(&application.WebviewWindowOptions{
        Title:  "Hello",
        Width:  1024,
        Height: 768,
    })
    app.Run()
}
```

Под капотом это:

1. Запускает `internal.runtime.*`
2. Настраивает `assetserver`
3. Регистрирует привязки, сгенерированные `internal.generator`
4. Переходит в основной поток ОС

## `examples/` – Живые спецификации

Каждая подпапка представляет собой **самодостаточное приложение**, демонстрирующее одну функцию.  
Шаблоны, которые вы увидите в реальном коде:

* Привязка сервисов (`examples/binding/`)  
* Многос оконность (`examples/window/`)  
* Системный трей (`examples/systray-*`)  
* Упаковка (`examples/file-association/`)  

Если сомневаетесь, начните с них и углубляйтесь в реализацию.

---

## `templates/` – Шаблоны для генерации

`internal/templates` содержит **базовые шаблоны** (структура Go) и **фронтенд-обёртки**
(React, Svelte, Vue, …).  
При выполнении `wails3 init -t react` CLI:

1. Копирует общие файлы Go из `_common`
2. Объединяет выбранный фронтенд-пакет
3. Запускает `npm install` + `task deps`

Редактирование шаблонов **не влияет** на существующие приложения, только на будущие `init`.

---

## `tasks/` – Автоматизация релизов

Taskfile-файлы обёртывают сложную кросс-компиляцию, обновление версий и генерацию changelog.  
Они вызываются программно из `internal/commands/task.go`, поэтому одна и та же логика используется как в **CLI**, так и в **CI**.

## Взаимодействие компонентов

```d2
direction: down
CLI: "wails3 CLI"
Generator: "internal/generator"
AssetDev: "assetserver (dev)"
Packager: "internal/packager"
AppRuntime: {
  label: "Среда выполнения приложения"
  ApplicationPkg: "pkg.application"
  InternalRuntime: "internal.runtime"
  OSAPIs: "API ОС"
}
CLI -> Generator: "build / generate"
CLI -> AssetDev: "dev"
CLI -> Packager: "package"
Generator -> ApplicationPkg: "bindings"
ApplicationPkg -> InternalRuntime
InternalRuntime -> OSAPIs
ApplicationPkg -> AssetDev
```

*CLI → generator → runtime* образует основной путь от **исходного кода** до **работающего десктопного приложения**.

## Советы по ориентации

| Нужно понять… | Посмотрите на… |
|---------------------|----------|
| Платформо-специфичные обёртки | `internal/runtime/*_DARWIN.go`, `*_windows.go`, … |
| Протокол моста | `pkg/application/messageprocessor_*.go` |
| Работу с ресурсами | `internal/assetserver`, `v3/templates/base/assets` |
| Процесс упаковки | `internal/commands/appimage.go`, `internal/packager` |
| Движок шаблонов | `internal/templates/templates.go` |
| Статический анализ | `internal/generator/*` |
