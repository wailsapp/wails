Надежные десктопные фреймворки требуют надежного тестирования.  
Wails v3 использует **многоуровневую стратегию**:

| Уровень | Цель | Инструменты |
|-------|------|---------|
| Модульные тесты (Unit tests) | Быстрая обратная связь для изолированных функций | `go test ./...` |
| Интеграционные / Примеры тестов | Проверка целостности функций между пакетами | Примеры приложений + headless проверки |
| Обнаружение гонок (Race detection) | Выявление гонок данных в рантайме и мостах | `go test -race`, `wails3 dev -race` |
| CI матрица | Проверка кросс-платформенности на каждом PR | GitHub Actions |

В этом документе объясняется, **где находятся тесты**, **как их запускать** и **что происходит в CI**.

---

## 1. Соглашения о структуре директорий

```
v3/
├── internal/.../_test.go     # Модульные тесты для внутренних пакетов
├── pkg/.../_test.go          # Тесты публичного API
├── examples/...              # Примеры приложений также служат интеграционными тестами
├── tasks/events/...          # Тесты сгенерированного кода запускаются в CI
└── pkg/application/RACE.md   # Руководство по тестам гонок и известные безопасные подавления
```

Рекомендации:

* **Храните модульные тесты рядом с кодом** (`foo.go` ↔ `foo_test.go`)
* Используйте **стиль black-box** для пакетов `pkg/` (`package application_test`)
* Интеграционные помощники находятся в `internal/testutil/` (общие моки, фикстуры)

## 2. Модульные тесты (Unit Tests)

### Написание тестов

```go
func TestColourParseHex(t *testing.T) {
    c, err := colour.Parse("#FF00FF")
    require.NoError(t, err)
    assert.Equal(t, 255, c.R)
}
```

Рекомендации:

* Используйте [`stretchr/testify`](https://github.com/stretchr/testify) – уже импортирован в vendor.
* Следуйте **табличному стилю** (table-driven) для ветвлений.
* При необходимости скрывайте особенности платформ за тегами сборки (`_test_windows.go` и т.д.).

### Локальный запуск

```bash
cd v3
go test ./... -cover
```

Быстрый запуск через Taskfile:

```
task test          # то же самое, что выше + vet + lint
```

## 3. Интеграционные и примерные тесты

Каждая папка под `v3/examples/` — это **самодостаточное приложение**.  
Три техники превращают их в тесты:

| Техника | Файл | Что проверяется |
|-----------|------|----------------|
| `go test ./...` внутри примера | `*_test.go` запускает `wails dev` в headless режиме | Сборка и запуск |
| Золотые тесты CLI | `internal/commands/appimage_test.go` | Хэши сгенерированных артефактов |
| Скриптовые e2e | `tasks/events/generate.go` | Генерирует Go код, который запускает пример и проверяет логи рантайма |

Запустите все вместе:

```
task integration
```

> Headless режим использует **Xvfb** на Linux раннерах; macOS и Windows запускаются свернутыми.

## 4. Обнаружение гонок (Race Detection)

Гонки данных фатальны для GUI рантаймов.

### Выделенная документация

Смотрите `v3/pkg/application/RACE.md` для:

* Известных безопасных гонок и обоснования их подавления
* Как интерпретировать стек-трейсы, пересекающие границы Cgo

### Локальный набор тестов гонок

```
go test -race ./...
```

или

```
wails3 dev -race
```

*Последний вариант пересобирает рантайм с флагами race и запускает демонстрационное окно.  
Закройте окно; любые гонки будут выведены после выхода.*

CI запускает набор тестов гонок на **linux/amd64** (самый быстрый) для каждого PR.

## 5. Рабочие процессы GitHub Actions

### 5.1 `build-and-test-v3.yml`

Расположение: `.github/workflows/build-and-test-v3.yml`

Ключевые особенности:

* **Матрица** по `os: [ubuntu-latest, windows-latest, macos-latest]`
* Кэширование Go и npm зависимостей для скорости
* Шаги:
  1. `setup-go` @ 1.25
  2. `go vet ./...`
  3. `go test ./... -v -coverprofile=cover.out`
  4. Сборка CLI + примера приложения (`wails3 build -skip-package`)
  5. Загрузка покрытия в Codecov
* Условный **race** job на Ubuntu:
  ```yaml
  - name: Race Detector
    if: matrix.os == 'ubuntu-latest'
    run: go test -race ./...
  ```

### 5.2 Статический анализ

* `semgrep.yml` – сканирование безопасности/линтинга (`.github/workflows/semgrep.yml`)
* `qodana.yaml` – облачная инспекция JetBrains Qodana (опционально в PR)

### 5.3 Пайплайн релизов

`runtime.yml` собирает и публикует JS рантайм в npm при пуше тега; вне области тестирования, но упоминается в матрице сборки.

## 6. Локальное соответствие CI

Запустите **точный** набор задач CI через Taskfile:

```
task ci
```

Он выполняет:

1. `lint` (golangci-lint, govet)
2. `test` (модульные + гонки)
3. `integration`
4. `build` (для хост-платформы)

---

## 7. Устранение неполадок падающих тестов

| Симптом | Вероятная причина | Решение |
|---------|--------------|-----|
| **Гонка в `runtime_darwin.go`** | Забыли заблокировать `mainthread.Call` | Используйте помощник `mainthread.Sync` |
| **Зависание Windows headless** | Консольное окно ждет ввода | Передайте `-verbose` и убедитесь, что диалоги не открываются |
| **Ошибка сборки примера в CI** | Шаблон изменился без обновления зависимостей примера | `task update-examples` перегенерирует lock-файлы |
| **Ошибки Coverpkg** | Интеграционный тест импортирует `main` | Переключитесь на теги сборки `//go:build integration` |

---

## 8. Добавление новых тестов

1. **Модульные** – создайте `*_test.go`, запустите `go test ./...`
2. **Интеграционные** – обновите или добавьте пример приложения + тестовый хост
3. **CI** – закоммитьте; рабочий процесс автоматически обнаружит тесты  
   Если тест специфичен для ОС, добавьте теги пропуска:

```go
//go:build !windows
```

Качество не является второстепенной мыслью в Wails v3.  
С помощью модульных тестов, интеграционных наборов, обнаружения гонок и кросс-платформенной CI
матрицы вы можете вносить вклад с уверенностью, зная, что ваши изменения проходят проверку на каждом
из поддерживаемых ОС.  
Удачного тестирования!
