## Обзор

Каждое приложение Wails поставляется в виде **одного нативного исполняемого файла**, который объединяет:

1. Ваш бэкенд на *Go*  
2. Веб-фронтенд (HTML + JS + CSS)

**Сервер ресурсов** (Asset Server) — это связующее звено, которое делает это возможным.  
Он имеет **два режима работы**, которые выбираются на этапе компиляции с помощью тегов сборки Go:

| Режим | Тег | Назначение |
|------|-----|---------|
| **Разработка** | `//go:build dev` | Быстрая итерация с горячей перезагрузкой |
| **Производство**  | `//go:build !dev` | Нулевые зависимости, встроенные ресурсы |

Реализация находится в  
`v3/internal/assetserver/` с четким разделением файлов:

```
assetserver_dev.go        # ⬅️ сервер разработки во время выполнения
assetserver_production.go # встроенный сервер
assetserver_darwin.go     # помощники, специфичные для ОС (те же для linux/windows)
asset_fileserver.go       # общая логика статического файлового сервера
content_type_sniffer.go   # определение типа MIME
ringqueue.go              # крошечная LRU для кэша mime
```

## Режим разработки

### Жизненный цикл

1. `wails3 dev` запускается и **запускает ваш сервер разработки фронтенда**  
   (Vite, SvelteKit, React-SWC …), выполняя задачу, определенную в
   `build/Taskfile.yml` (обычно `npm run dev`).
2. Wails запускает **Сервер ресурсов разработки**, прослушивающий `localhost:<random>` и
   сообщает рантайму Go загрузить `http://<host>:<VITE_PORT>` как URL окна.
3. Входящие запросы обрабатываются `assetserver_dev.go`:

   ```
   ┌─────────┐  /runtime/...          ┌─────────────┐
   │ Browser │ ── нативный мост  ───▶ │   Runtime   │
   ├─────────┤                       └─────────────┘
   │   JS    │  / (index.html)        прокси / -> Vite
   └─────────┘ ◀─────────────┐
                 AssetServer │
                             ▼
                      ┌────────────┐
                      │  Vite Dev  │
                      │   Server   │
                      └────────────┘
   ```

4. Статические файлы (`/assets/logo.svg`) **обслуживаются напрямую с диска** через
   `asset_fileserver.go` (для скорости), а все неизвестное **проксируется** на
   сервер разработки фреймворка, обеспечивая *мгновенную* замену модулей (hot-module replacement).

### Возможности

* **Живая перезагрузка** – Vite/Snowpack/… внедряет WebSocket HMR; Wails только должен
  проксировать его.
* **Поддержка Source Map** – поскольку ресурсы не упакованы, инструменты разработчика браузера
  отображают ошибки обратно в исходный код.
* **Перекомпиляция Go не требуется** – пересобирается только фронтенд; код Go продолжает работать, пока
  вы не измените файлы `.go`.

### Переключение фреймворков

Прокси разработки **независим от фреймворка**:

* Корневой файл `Taskfile.yml` устанавливает две переменные окружения:
  `APP_NAME` & `WAILS_VITE_PORT`.
* Taskfiles для каждого шаблона передают эти переменные перед запуском своих серверов разработки.
* `assetserver_dev.go` просто проксирует на эту цель.

Добавьте новый шаблон → определите его задачу разработки → Сервер ресурсов просто работает.

## Режим производства

Когда вы запускаете `wails3 build`, конвейер:

1. Выполняет **сборку фронтенда для производства** (`npm run build`), создавая
   `/frontend/dist/**`.
2. **Встраивает** эту папку в файловую систему `go:embed` на этапе компиляции (см.
   сгенерированный файл `bundled_assetserver.go`).
3. Компилирует двоичный файл Go с `-tags production` (неявно).

### Обработка запросов

```go
func (h *Handler) ServeHTTP(w http.ResponseWriter, r *http.Request) {
    // 1. Попробовать встроенные статические ресурсы (точный путь)
    // 2. Резервный вариант: index.html для маршрутизации SPA
    // 3. Определить content-type, если расширение неизвестно
    // 4. Установить заголовки сильного кэширования
}
```

* **Определение MIME** – Если инструмент сборки создал файлы без расширений (например,
  `/assets/manifest`), `content_type_sniffer.go` просматривает первые 512 байт и
  кэширует результат в крошечной LRU без блокировок.
* **Кэширование кольцевого буфера** – Часто используемые ресурсы (логотип, CSS) хранятся
  в памяти на протяжении всего времени работы приложения, устраняя необходимость в поиске на диске или во встроенной ФС.
* **Заголовки безопасности** – Запрещает навигацию `file://`, включает `nosniff`.

Поскольку все встроено, выпускаемый двоичный файл не имеет **внешних
зависимостей** (даже в Windows).

## Связь Dev ↔ Prod

Оба режима предоставляют **общий публичный интерфейс**:

```go
type AssetServer interface {
    URL()        string // dev: http://localhost:34115, prod: wails://app
    Open() error // начать прослушивание
    Close()      // корректное завершение работы
}
```

`pkg/application` с удовольствием использует ту реализацию, которая была скомпилирована, что означает,
что **код вашего приложения не меняется** между `dev` и `build`.

## Как фреймворки фронтенда интегрируются

### Шаблоны

Каждый официальный шаблон (React, Vue, Svelte, Solid…) содержит:

* `build/Taskfile.yml`
* `frontend/vite.config.ts` (или эквивалент)

Они экспортируют несколько задач, включая:

| Задача | Назначение |
|------|---------|
| `dev:frontend`   | Запускает сервер разработки фреймворка на **случайном свободном порту** и выводит его в stdout (`PORT=5173`). |
| `build:frontend` | Создает статические ресурсы в `dist/` + манифест для обхода кэша. |

`internal/commands/dev.go` устанавливает переменную окружения `FRONTEND_DEVSERVER_URL`, используя жестко заданный `localhost` в качестве `host` и выведенную переменную окружения `VITE_PORT` в качестве `port`, и запускает **Сервер ресурсов разработки**.

Фреймворки остаются полностью отделенными от Go:

* Нет необходимости импортировать Wails JS SDK на этапе сборки – рантайм внедряет его при создании окна.
* Любой фреймворк с HTTP-сервером разработки может подключиться.

---

### Расширение / Настройка

Нужны пользовательские заголовки, аутентификация или gzip?

1. Реализуйте `type Middleware func(http.Handler) http.Handler`
2. Зарегистрируйте через `internal/assetserver/options.go`
3. Для prod не забудьте добавить тот же middleware в `assetserver_production.go`.

## Ключевые исходные файлы

| Файл | Роль |
|------|------|
| `assetserver_dev.go` | Обратный прокси + файловый сервер на диске |
| `assetserver_production.go` | Обработчик встроенной ФС |
| `options.go` | Структура конфигурации, распарсенная из `pkg/options/assetserver` |
| `build_dev.go` / `build_production.go` | Обертки тегов сборки, выбирающие правильную реализацию |
| `bundled_assetserver.go` | Сгенерированные данные встраивания (присутствует только в сборках для производства) |

## Подводные камни и отладка

* **Белый экран в Prod** – обычно маршрутизация SPA: убедитесь, что `History API Fallback`
  включен в dev и срабатывает fallback `index.html` в prod.
* **404 в Dev** – несовпадение `FRONTEND_DEV_PORT`; запустите с
  `WAILSDEV_VERBOSE=1`, чтобы печатать каждый проксируемый запрос.
* **Большие ресурсы** – они встраиваются; рассмотрите
  [`assetserver.WithExternalDir("/path")`](https://pkg.go.dev) для загрузки с диска.

Теперь вы знаете, как **Сервер ресурсов** Wails подает ваш веб-код в нативное
окно как в **режиме разработки**, так и в **режиме производства**.  
Освойте этот слой, и вы сможете отлаживать проблемы загрузки, добавлять middleware или даже
с уверенностью заменять весь стек инструментов фронтенда.
