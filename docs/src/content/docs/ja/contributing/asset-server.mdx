## 概要

すべての Wails アプリケーションは、以下の要素を結合した**単一のネイティブ実行ファイル**をリリースします。

1. *Go* バックエンド  
2. *Web* フロントエンド (HTML + JS + CSS)

**アセットサーバー**は、これを可能にする接着剤の役割を果たします。  
アセットサーバーには、Go のビルドタグによってコンパイル時に選択される**2つの動作モード**があります。

| モード | タグ | 目的 |
|------|-----|---------|
| **開発** | `//go:build dev` | ホットリロードによる高速な反復処理 |
| **本番**  | `//go:build !dev` | 依存関係ゼロ、埋め込みアセット |

実装は `v3/internal/assetserver/` にあり、ファイルの分割が明確です。

```
assetserver_dev.go        # ⬅️ ランタイム開発サーバー
assetserver_production.go # ⬅️ 埋め込みサーバー
assetserver_darwin.go     # OS固有のヘルパー (linux/windows でも同様)
asset_fileserver.go       # 共有静的ファイルロジック
content_type_sniffer.go   # MIME タイプ検出
ringqueue.go              # MIME キャッシュ用の小さな LRU
```

## 開発モード

### ライフサイクル

1. `wails3 dev` が起動し、`build/Taskfile.yml` (通常は `npm run dev`) で定義されたタスクを実行して**フロントエンドの開発サーバー** (Vite, SvelteKit, React-SWC …) を起動します。
2. Wails は `localhost:<ランダムポート>` でリッスンする**開発用アセットサーバー**を起動し、Go ランタイムに `http://<ホスト>:<VITE_PORT>` をウィンドウ URL として読み込むよう指示します。
3. 着信リクエストは `assetserver_dev.go` によって処理されます。

   ```
   ┌─────────┐  /runtime/...          ┌─────────────┐
   │ Browser │ ── native bridge  ───▶ │   Runtime   │
   ├─────────┤                       └─────────────┘
   │   JS    │  / (index.html)        proxy / -> Vite
   └─────────┘ ◀─────────────┐
                 AssetServer │
                             ▼
                      ┌────────────┐
                      │  Vite Dev  │
                      │   Server   │
                      └────────────┘
   ```

4. 静的ファイル (`/assets/logo.svg`) は、速度のために `asset_fileserver.go` 経由で**ディスクから直接提供**されます。一方、不明なリクエストはフレームワークの開発サーバーに**プロキシ**され、*即時の*ホットモジュールリプレイスメントを実現します。

### 機能

* **ライブリロード** – Vite/Snowpack/… が HMR WebSocket を注入します。Wails はそれをプロキシするだけです。
* **ソースマップサポート** – アセットがバンドルされていないため、ブラウザの開発ツールはエラーを元のソースにマッピングできます。
* **Go の再コンパイル不要** – フロントエンドのみが再ビルドされ、`.go` ファイルを変更するまで Go コードは実行されたままになります。

### フレームワークの切り替え

開発プロキシは**フレームワークに依存しません**。

* ルートの `Taskfile.yml` タスクファイルは、2 つの環境変数 `APP_NAME` と `WAILS_VITE_PORT` を注入します。
* 各テンプレートの Taskfile は、開発サーバーを実行する前にこれらの変数を出力します。
* `assetserver_dev.go` は単にそのターゲットにプロキシします。

新しいテンプレートを追加 → 開発タスクを定義 → アセットサーバーはそのまま動作します。

## 本番モード

`wails3 build` を実行すると、パイプラインは以下の手順で動作します。

1. フロントエンドの**本番ビルド** (`npm run build`) を実行し、`/frontend/dist/**` を生成します。
2. そのフォルダをコンパイル時に `go:embed` ファイルシステムに**埋め込み**ます (生成されたファイル `bundled_assetserver.go` を参照)。
3. Go バイナリを `-tags production` (暗黙的) でコンパイルします。

### リクエスト処理

```go
func (h *Handler) ServeHTTP(w http.ResponseWriter, r *http.Request) {
    // 1. 埋め込み静的アセット (正確なパス) を試す
    // 2. SPA ルーティング用に index.html にフォールバック
    // 3. 拡張子が不明な場合、コンテンツタイプを検出
    // 4. 強力なキャッシュヘッダーを設定
}
```

* **MIME 検出** – ビルドツールが拡張子のないファイル (例: `/assets/manifest`) を生成した場合、`content_type_sniffer.go` が最初の 512 バイトを調べ、その結果を小さなロックフリーの LRU にキャッシュします。
* **リングキューキャッシング** – 頻繁にアクセスされるアセット (ロゴ、CSS) は、アプリの存続期間中メモリ内に保持され、ディスクや埋め込み FS へのルックアップを不要にします。
* **セキュリティヘッダー** – `file://` によるナビゲーションを禁止し、`nosniff` を有効にします。

すべてが埋め込まれているため、リリースされるバイナリには**外部依存関係がありません** (Windows でも同様です)。

## 開発と本番の橋渡し

両方のモードは**同じ公開インターフェース**を公開します。

```go
type AssetServer interface {
    URL()        string // 開発: http://localhost:34115, 本番: wails://app
    Open() error // リッスンを開始
    Close()      // グraceful シャットダウン
}
```

`pkg/application` は、コンパイル時に組み込まれた実装を喜んで使用するため、**アプリケーションコードは `dev` と `build` の間で変更されません**。

## フロントエンドフレームワークの統合方法

### テンプレート

各公式テンプレート (React, Vue, Svelte, Solid…) には以下が含まれています。

* `build/Taskfile.yml`
* `frontend/vite.config.ts` (または同等のもの)

これらは以下のいくつかのタスクをエクスポートします。

| タスク | 目的 |
|------|---------|
| `dev:frontend`   | フレームワークの開発サーバーを**ランダムな空きポート**で起動し、そのポート番号を標準出力に出力します (`PORT=5173`)。 |
| `build:frontend` | キャッシュ busting 用のマニフェストとともに、静的アセットを `dist/` に生成します。 |

`internal/commands/dev.go` は、`host` としてハードコードされた `localhost` と、出力された `VITE_PORT` 環境変数を `port` として使用して `FRONTEND_DEVSERVER_URL` 環境変数を設定し、**開発用アセットサーバー**を起動します。

フレームワークは Go から完全に分離されています。

* ビルド時に Wails JS SDK をインポートする必要はありません – ランタイムがウィンドウ作成時にそれを注入します。
* HTTP 開発サーバーを持つ任意のフレームワークをプラグインできます。

## 拡張 / カスタマイズ

カスタムヘッダー、認証、または gzip が必要ですか？

1. `type Middleware func(http.Handler) http.Handler` を実装します。
2. `internal/assetserver/options.go` 経由で登録します。
3. 本番環境では、`assetserver_production.go` にも同じミドルウェアを追加することを忘れないでください。

## 主要なソースファイル

| ファイル | 役割 |
|------|------|
| `assetserver_dev.go` | リバースプロキシ + ディスクファイルサーバー |
| `assetserver_production.go` | 埋め込み FS ハンドラー |
| `options.go` | `pkg/options/assetserver` から解析される構成構造体 |
| `build_dev.go` / `build_production.go` | 正しい実装を選択するビルドタグラッパー |
| `bundled_assetserver.go` | 生成された埋め込みデータ (本番ビルドでのみ存在) |

## 注意点とデバッグ

* **本番での白画面** – 通常は SPA ルーティングの問題です。開発で `History API Fallback` が有効になっていること、および本番で `index.html` のフォールバックが機能していることを確認してください。
* **開発での 404** – `FRONTEND_DEV_PORT` の不一致です。`WAILSDEV_VERBOSE=1` を付けて実行すると、プロキシされたすべてのリクエストが出力されます。
* **大きなアセット** – これらは埋め込まれます。ディスクから読み込むには [`assetserver.WithExternalDir("/path")`](https://pkg.go.dev) を検討してください。

---

これで、Wails **アセットサーバー**が、**開発**および**本番**の両方で Web コードをネイティブウィンドウにどのように供給するかを理解しました。  
このレイヤーをマスターすれば、読み込みの問題のデバッグ、ミドルウェアの追加、あるいは完全に異なるフロントエンドツールチェーンへの置き換えを自信を持って行うことができます。
