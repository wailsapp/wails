> "Bindings" are the **type-safe contract** that lets you write:

```go
msg, err := chatService.Send("Hello")
```

in Go *and*

```ts
import { Send } from "../bindings/github.com/you/yourapp/services/chatservice";

const msg = await Send("Hello");
```

in TypeScript **without writing any IPC glue by hand**.
This document breaks down *how* that happens, from **static analysis** at
build time, through **code generation**, to the **runtime bridge** that moves
bytes over the WebView.

> See [`contributing/architecture/bindings.mdx`](./architecture/bindings) for
> the authoritative deep-dive on the generator pipeline — this page is a
> contributor-focused overview.

---

## 1. 30-Second Overview

| Stage | Component | Output |
|-------|-----------|--------|
| **Collect / Analyse** | `internal/generator/collect/`, `internal/generator/analyse.go` | In-memory model of exported Go services, methods, params, return types, models |
| **Generation** | `internal/generator/render/templates/*.tmpl` (`service.{js,ts}.tmpl`, `models.{js,ts}.tmpl`, `index.tmpl`, `eventcreate.js.tmpl`, `eventdata.d.ts.tmpl`, `newline.tmpl`) | Per-service ES modules under `frontend/bindings/<full Go import path>/...` |
| **Runtime** | `pkg/application/messageprocessor*.go` + the embedded JS runtime under `internal/runtime/desktop/@wailsio/runtime/src/` (`calls.ts`, `events.ts`, …) | Call/event messages over the WebView's native bridge |

The flow is orchestrated by the `wails3 generate bindings` command, which
drives `generator.Generate` (defined in `internal/generator/generate.go`)
over a set of Go packages.

```
wails3 generate bindings
        │
        ▼
internal/generator/generate.go: Generator.Generate(patterns...)
        │
        ├── internal/generator/collect/   // load.go, collector.go, service.go, model.go, …
        ├── internal/generator/analyse.go // semantic checks
        └── internal/generator/render/    // template execution → frontend/bindings/**
```

## 2. Static Analysis

### Entry Point

```
internal/generator/generate.go        // Generator + Generate(patterns…)
internal/generator/analyse.go         // semantic validation
internal/generator/collect/load.go    // go/packages loader
internal/generator/collect/collector.go
```

The collector pass walks every loaded package and records:

* `collect.ServiceInfo` — one per exported, bound Go struct.
* `collect.ServiceMethodInfo` / `collect.MethodInfo` — per-method signature info
  (name, params, results, error position, receiver, doc).
* `collect.ModelInfo` / `collect.StructInfo` — emitted as TS / JS models.
* Directive comments such as `//wails:inject`, `//wails:include`,
  `//wails:internal`, `//wails:ignore`, `//wails:id <hex>` (see
  `internal/generator/collect/directive.go`).

Unsupported types produce a generator error so mistakes surface at build time,
not at runtime.

### Model Identifiers

The runtime call envelope identifies a method by a **deterministic FNV-1a hash**
of its fully-qualified name (`pkg.Struct.Method`). You'll see this as
`$Call.ByID(<numeric-id>, …)` in generated bindings or `$Call.ByName("pkg.Struct.Method", …)`
when generation runs with `-names`.

## 3. Code Generation

### Templates

`internal/generator/render/templates/`:

| Template | Purpose |
|----------|---------|
| `service.js.tmpl` | One JS module per bound service |
| `service.ts.tmpl` | TypeScript companion (with `-ts`) |
| `models.js.tmpl` | Model class output (per-package) |
| `models.ts.tmpl` | Model `.d.ts` output (per-package) |
| `index.tmpl` | Per-package `index.{js,ts}` barrel re-exports |
| `eventcreate.js.tmpl` / `eventdata.d.ts.tmpl` | Event constructor / payload typings |
| `newline.tmpl` | Trailing-newline normaliser |

Output goes under `frontend/bindings/<full Go import path>/...` — for example
a service defined in `github.com/you/yourapp/services/chat` lands under
`frontend/bindings/github.com/you/yourapp/services/chat/`. There is no
`frontend/src/wailsjs/` directory in v3.

### JavaScript Output

Generated bindings are ES modules that import the runtime helpers from
`/wails/runtime.js`:

```js
import { Call as $Call, Create as $Create } from "/wails/runtime.js";

/**
 * @param {string} msg
 * @returns {Promise<string> & { cancel(): void }}
 */
export function Send(msg) {
    return $Call.ByID(2042131923, msg);
}
```

When generation runs with `-names`, `$Call.ByName("pkg.Struct.Method", ...)`
is emitted instead — always **fully-qualified**, never just `"Method"`.

Generated model classes use a `$$source` constructor pattern with per-field
`if (!("X" in $$source))` defaults, quoted field names, and a
`static createFrom(...)` that runs `JSON.parse` on string inputs.

### Type Mapping Highlights

Verified against `internal/generator/render/`:

| Go | TypeScript |
|----|------------|
| `map[string]V` | `{ [_: string]: V }` |
| `map[K]V` (non-string `K`) | `{ [_ in K]?: V }` (not `Map<K, V>`, not `Record<K, V>`) |
| `[]byte` | `Uint8Array` |
| `time.Time` | `string` (JSON ISO 8601) |
| `error` (return position) | rejected promise |

### Reflection Note

`pkg/application/bindings.go` is **hand-written** and uses `reflect` to drive
method dispatch from a `BoundMethod` registry. Don't read older claims of
"zero reflection at runtime" too literally — the generator avoids reflection,
but the runtime dispatcher uses it.

## 4. Runtime Invocation Protocol

### JavaScript Side

```ts
import { Call } from "/wails/runtime.js";

await Call.ByID(0x7a1201d3 /* ChatService.Send */, "Hello");
// or, with -names:
await Call.ByName("chatservice.ChatService.Send", "Hello");
```

The runtime helpers live in
`internal/runtime/desktop/@wailsio/runtime/src/calls.ts` (call dispatch),
`events.ts` (events), and friends — there is no `invoke.ts` or `errors.ts`
in this tree. The exact wire envelope is encoded by `calls.ts` on the JS
side and decoded by `pkg/application/messageprocessor_call.go` on the Go
side; consult those two files together when debugging the bridge.

### Go Side

1. `pkg/application/messageprocessor_call.go` receives the call message.
2. Looks up the bound method by ID or name in
   `pkg/application/bindings.go` (driven by `reflect`).
3. Invokes the bound method and serialises `{result, error}` back to JS.

### Error Mapping

| Go | JavaScript |
|----|------------|
| `error == nil` | `Promise` resolves with result |
| `error != nil` | `Promise` rejects with an `Error` whose `message` is the Go error string |

## 5. Calling JavaScript from Go

The binding generator is one-way (Go methods exposed to JS). For Go → JS
communication, use the event bus or run JS in a window:

```go
app.Event.Emit("chat:new-message", msg)
window.ExecJS(`window.dispatchEvent(new CustomEvent("ping"))`)
```

On the JS side, subscribe with `Events.On(name, cb)` from
`/wails/runtime.js`.

## 6. Extending & Troubleshooting

### Unsupported Type Error

```
error: field "Client" uses unsupported type: chan struct{}
```

→ wrap the channel behind a method API, or mark the field with
`//wails:internal` so the generator skips it.

### Stale Bindings

Generated output is overwritten on every `wails3 generate bindings` /
`wails3 dev` / `wails3 build`. If IDE intellisense shows stale stubs, delete
`frontend/bindings/` and re-run the generator. The `-clean` flag (default
`true` in current builds) wipes the bindings directory before each run.

### Performance Tips

* Avoid streaming large byte slices over the bridge — serve them via the
  asset server instead.
* Batch multiple quick calls into one method when latency matters.
* Prefer value receivers for small parameter structs to reduce allocations.

## 7. Key Files Map

| Concern | File |
|---------|------|
| Generator orchestration | `internal/generator/generate.go` |
| Semantic checks | `internal/generator/analyse.go` |
| Collection (services, methods, models) | `internal/generator/collect/{service,method,model,struct,package}.go` |
| Render templates | `internal/generator/render/templates/*.tmpl` |
| Generated bindings location | `frontend/bindings/<full Go import path>/...` |
| Go side dispatcher | `pkg/application/bindings.go`, `messageprocessor_call.go` |
| JS runtime | `internal/runtime/desktop/@wailsio/runtime/src/{calls,events,index}.ts` |

Keep this cheat-sheet handy when you trace a bridge bug.

## 8. Recap

1. **Collector** scans your Go code → in-memory semantic model.
2. **Templates** emit per-service ES modules and per-package model/index files.
3. **Message Processor** dispatches calls on the Go side via the bindings registry.
4. **JS Runtime** wraps it all in idiomatic promises with cancellation.

All without you writing a single line of IPC boilerplate. That's the Wails v3
binding system. Go forth and bind!
