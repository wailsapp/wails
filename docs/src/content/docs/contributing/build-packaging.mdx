`wails3 build` is intentionally **thin**: it's a Taskfile wrapper that forwards
extra build tags to the host project's `build` task. The heavy lifting lives in
the project's own `build/Taskfile.yml` (generated by `wails3 init`), in
`internal/commands/build-assets.go` (which manages bake-time assets), and in
`internal/packager` (Linux nfpm packaging) plus `internal/commands/{appimage,
msix,dmg/,dot_desktop}.go` (per-platform installers).

This page covers:

1. The actual CLI entry-points
2. The Taskfile-driven build flow
3. Asset bake & build-info injection
4. Packaging back-ends per platform
5. Customising the pipeline
6. Troubleshooting

---

## 1. Actual CLI Entry-Points

```
wails3 build       → internal/commands.Build       (in task_wrapper.go)
wails3 package     → internal/commands.Package     (in task_wrapper.go)
wails3 generate build-assets → GenerateBuildAssets (in build-assets.go)
wails3 update build-assets   → UpdateBuildAssets   (in build-assets.go)
wails3 tool buildinfo        → BuildInfoOptions    (in tool_buildinfo.go)
wails3 tool package          → internal/packager   (nfpm wrapper)
wails3 generate .desktop     → in dot_desktop.go
```

`internal/commands/task_wrapper.go`:

```go
func Build(buildFlags *flags.Build, otherArgs []string) error {
    // forwards --tags / EXTRA_TAGS, then defers to a Taskfile target
    return wrapTask("build", otherArgs)
}
```

`flags.Build` exposes a **single** flag — `--tags` (forwarded as
`EXTRA_TAGS=`). There are **no** `-platform`, `-o`, `-skipbindings`,
`-skip-package`, `-package`, `-ldflags`, `-verbose`, `-debug`, `-devbuild`,
`-icon`, or `-clean` flags on `wails3 build`. Cross-compilation, output paths,
icons, etc. are configured in **`Taskfile.yml`**, **`build/config.yml`**, and
the supporting `wails3 generate icons` / `wails3 generate build-assets`
commands.

`build/build.json` is **not** part of v3 — the config is `Taskfile.yml` plus
`build/config.yml`.

## 2. Taskfile-Driven Build Flow

A freshly initialised project ships a `build/Taskfile.yml` with namespaces
roughly:

| Namespace | Tasks (selection) |
|-----------|-------------------|
| `darwin:` | `build`, `build:universal`, `package`, `run`, `dev` |
| `windows:` | `build`, `package`, `run`, `dev` |
| `linux:` | `build`, `package`, `run`, `dev` |
| `common:` | `update:build-assets`, `generate:icons`, `generate:syso` |

`wails3 build` invokes the host OS's `build` namespace by default; the
project's Taskfile in turn shells out to `go build` with the host-specific
flags. Building for another OS means running its task directly (e.g.
`wails3 task darwin:build:universal`), not passing a flag to `wails3 build`.

The default output directory is **`bin/<APP_NAME>`** (no `build/bin/`
prefix).

## 3. Bake-Time Assets and Build Info

| Concern | File |
|---------|------|
| Build-asset generation/update | `internal/commands/build-assets.go` |
| Build info printer (CLI: `wails3 tool buildinfo`) | `internal/commands/tool_buildinfo.go` — prints information; it is **not** an `ldflags` injector |
| Production stub | `internal/assetserver/build_production.go` — `//go:build production` |
| Frontend bundles | embedded via `//go:embed` in the application's own package (e.g. next to `main.go`) |
| Windows resources (`.syso`) | `internal/commands/syso.go` — generates `rsrc_windows_<arch>.syso` |
| Windows MSIX | `internal/commands/msix.go` + `internal/commands/webview2/` |
| macOS DMG inputs | `internal/commands/dmg/` |
| Linux `.desktop` | `internal/commands/dot_desktop.go` |

The CLI does not auto-bake `bundled_assetserver.go` for your application —
`internal/assetserver/bundled_assetserver.go` is **hand-written** and wraps
the embedded JS runtime under `bundledassets/`.

## 4. Packaging Back-Ends

### Linux

Linux packaging is driven by **nfpm** (not `fpm`):

* `internal/packager/packager.go` wraps `github.com/goreleaser/nfpm/v2` and
  exposes `CreatePackageFromConfig(pkgType, configPath, output)` /
  `CreatePackageFromConfigWriter(...)`.
* Generated projects ship `myapp.DEB`, `myapp.RPM`, and `myapp.ARCHLINUX`
  nfpm-style configs under `internal/commands/` (used by `wails3 tool package`).
* AppImage generation lives in `internal/commands/appimage.go`, which calls
  `linuxdeploy` + `linuxdeploy-plugin-gtk` (the plugin ships at
  `internal/commands/linuxdeploy-plugin-gtk.sh`).

There is **no** `-package deb`/`rpm` flag on `wails3 build`. Use
`wails3 tool package` or the platform-specific Taskfile target.

### macOS

* `darwin:package` from the project Taskfile produces the `.app` bundle.
* DMG assets live under `internal/commands/dmg/`; a project can wrap the
  bundle into a DMG with `hdiutil` after `darwin:package` finishes (the
  Taskfile in newer templates includes a `dmg` helper).
* CFBundle identifiers, version, and copyright come from the `-product*`
  flags during `wails3 init` and from `build/config.yml`.

### Windows

* Windows packaging targets **MSIX** (not WiX/MSI). See
  `internal/commands/msix.go` and `internal/commands/webview2/` for the
  full workflow.
* There is **no** `internal/commands/packager.go` and **no**
  `internal/commands/windows_resources/` directory.
* Optional code-signing for the executable runs via `wails3 tool sign`
  (Authenticode) — see `internal/commands/sign.go`.

## 5. Customising the Pipeline

| Need | Approach |
|------|----------|
| Extra build tags | `wails3 build --tags myFeature,otherTag` |
| Linter / pre-build step | Add a task to `build/Taskfile.yml` and make the OS-specific `build` task depend on it |
| Cross-compile | Run the relevant OS task (e.g. `wails3 task linux:build`) — there is no `-platform` flag |
| Skip packaging | Just run the `build` task; `package` is separate |
| Custom packager | Drop a config under `internal/commands/myapp.*` and call `wails3 tool package` with `-config <file>` |
| Strip symbols | Edit the `darwin:/windows:/linux:` `build` task to pass `-ldflags "-s -w"` directly to `go build` — `wails3 build` itself has no `-ldflags` flag |

All Taskfile targets respect the env vars Wails publishes (`APP_NAME`,
`WAILS_VITE_PORT`, `FRONTEND_DEVSERVER_URL`, …) so custom tasks can rely on
them.

## 6. Troubleshooting

| Symptom | Likely Cause | Fix |
|---------|--------------|-----|
| **`ld: framework not found WebKit` (mac)** | Xcode CLI tools missing | `xcode-select --install` |
| **Blank window in prod build** | Frontend build failed or SPA routing | Check `frontend/dist/index.html` exists and your asset handler falls back to it |
| **MSIX packaging tools missing** | `WebView2` SDK / MSIX tools not installed | Run `wails3 task install:msix:tools` |
| **`linuxdeploy` not found** | Plugin missing on PATH | Install `linuxdeploy` + run `internal/commands/linuxdeploy-plugin-gtk.sh` via the CLI's auto-install step |

`wails3 build` has no `-verbose` flag. Set `TASK_X_VERBOSE=1` (Taskfile) or
inspect the task target directly to see the commands being executed.

## 7. Key Source Map

| Concern | File |
|---------|------|
| Build wrapper | `internal/commands/task_wrapper.go` (`Build`, `Package`, `SignWrapper`, `wrapTask`) |
| Build-asset generation | `internal/commands/build-assets.go` (`GenerateBuildAssets`, `UpdateBuildAssets`) |

| Windows MSIX | `internal/commands/msix.go`, `internal/commands/webview2/` |
| Windows resource generator | `internal/commands/syso.go` |
| macOS DMG assets | `internal/commands/dmg/` |
| `.desktop` generator | `internal/commands/dot_desktop.go` |
| Version constants | `internal/version/version.go` |

Keep this table handy when you trace a build failure.

---

You now have the full picture from **source code** to **installer**. The TL;DR
is that `wails3 build` itself is a thin wrapper — almost every customisation
happens in the project's `Taskfile.yml` / `build/config.yml`, or via the
explicit `wails3 generate …` / `wails3 tool …` subcommands. Happy shipping!
